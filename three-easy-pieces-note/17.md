# 17. 空き領域管理（Free-Space Management）

> 🎯 **この章を学ぶ理由**: `malloc()`/`free()`の内部で何が起きているかを理解する。メモリの断片化やアロケータの設計は、パフォーマンスチューニングやガベージコレクションの理解にもつながる。
> **前提知識**: 14章（メモリAPI）

この章では、可変サイズのメモリ割り当て要求（「10バイト欲しい」「50バイト欲しい」などバラバラな要求）を処理する際の**空き領域管理**について学ぶ。`malloc()`の大元となる仕組みだ。

最大の壁は**外部断片化（External Fragmentation）**だ。
> 💡 **外部断片化とは？（駐車場の例）**
> 駐車場全体で「合計3台分」の空きスペースがあるとする。しかし、それが離れ離れに1台分ずつ空いているため、3台分の長さを必要とする**大型バスが停められない**。
> メモリでも同じように、「合計すれば空きは足りるのに、連続した空きスペースがないせいで割り当てに失敗する」問題が起きる。

![](./images/fig17_1_1.PNG)

## 17.1 前提

- `malloc(size_t size)` と `free(void *ptr)` のインタフェースを想定
- `free()`にはサイズ引数がない → ライブラリがサイズを追跡する必要がある
- 一度ユーザに渡したメモリは移動できない（圧縮不可）

## 17.2 低レベルのメカニズム

### 分割（Splitting）と合体（Coalescing）

![](./images/fig17_1_2.PNG)

30バイトのヒープに、10バイトの空きチャンクが2つある場合を考える。これらをどう管理するか？「分割」と「合体」という2つの基本操作がある。

> 💡 **レストランの座席アレンジの例**

**分割（Splitting）**:
10人用の大きなテーブルに、3人のグループが来店したとする。アロケータはテーブルを「3人用（割り当て済み）」と「7人用（引き続き空き）」に**分割**して案内する。要求サイズぴったりでなくても、大きい空きを切り分けて対応する。

**合体（Coalescing）**:
客が帰って席が空いたとき、もし「隣の席」も空いているなら、間の仕切りを外して**合体**させ、さらに大きなグループ（例えば15人用）を案内できるようにする。これを怠ると、小さな空き席ばかりになり、大型バス（外部断片化）が停められなくなってしまう。

![](./images/fig17_1_6.PNG)

### 割り当て領域サイズの追跡

`free()`関数を呼ぶとき、プログラマは`free(ptr)`のようにポインタだけを渡し、「何バイト解放するか」を指定しない。では、OS（ライブラリ）はどうやって解放すべきサイズを知るのか？

> 💡 **クロークの荷物預かりタグ**
> 荷物を預けたとき、荷物に「サイズや持ち主を示すタグ」がくっつけられる。返却時は荷物の場所（ポインタ）さえ伝えれば、係員がタグを見てサイズを把握してくれる。

メモリチャンクの場合、ユーザーに渡すアドレスの**直前（手前）**に**ヘッダブロック**（タグ）を隠して配置している。

```c
typedef struct {
    int size;   // 確保したサイズ
    int magic;  // 不正防止の「サニティチェック（正気度チェック）」用マジックナンバー
} header_t;
```

![](./images/fig17_2.PNG)

`free(ptr)`が呼ばれると、ライブラリは`ptr`から少し手前に戻り、ヘッダからサイズを読み取る。実際に解放される領域は「ヘッダサイズ ＋ データサイズ」となる。

### 埋め込みフリーリスト

「どこが空いているか」のリスト（フリーリスト）をメモリのどこかに保存する必要があるが、`malloc`を作るために`malloc`を呼ぶことはできない。どうするか？

> 💡 **空室物件の入り口にある「次の空室」の看板**
> アロケータは、**「空いているメモリ領域自体」の中に、リスト構造（次の空き領域へのポインタとサイズ）を書き込む**。領域を使っている（割り当て済み）間はユーザーのデータが入り、空いている（解放済み）間はアロケータの管理用看板として再利用される。

```c
typedef struct node_t {
    int size;
    struct node_t *next;
} node_t;
```

![](./images/fig17_3.PNG)

割り当てと解放の流れ：

1. 最初は1つの大きな空きチャンク
2. 割り当てるたびにチャンクを分割し、フリーリストの空きサイズが縮小
3. 解放されたチャンクをフリーリストに再追加
4. 隣接する空きチャンクを合体

![](./images/fig17_4.PNG)

![](./images/fig17_5.PNG)

### ヒープの拡張

ヒープの空きがなくなったら、`sbrk`などのシステムコールでOSに追加メモリを要求できる。

## 17.3 基本戦略

あるサイズの要求が来たとき、「どの空きスペースを渡すか」を決める戦略にもトレードオフがある。洋服屋さんでシャツを探す状況に似ている。

### ベストフィット（Best Fit）

> **「一番無駄のないピッタリのサイズを探す」**

要求サイズ以上の空きチャンクの中で**最小のもの**を選ぶ。断片化を最小限に抑えようとするが、リスト全体を検索する必要があるため**非常に遅い**。

### ワーストフィット（Worst Fit）

> **「あえて一番ブカブカのサイズを選ぶ」**

**最大のチャンク**を選ぶ。大きな空きを分割すれば、残った部分もまだ十分な大きさがあるだろうという狙いだ。しかし、実際には細かな断片が散らばりやすく、ベストフィット同様に全体検索が必要で遅い。

### ファーストフィット（First Fit）

> **「手前から探して、入るサイズが見つかったら即決定」**

最初に見つかった十分大きなチャンクを選ぶ。**検索が早く済む**のが最大のメリット。ただし、リストの先頭付近が細かく分割されて散らかり「小さな空き」がたまりやすい。リストをアドレス順にソートしておくと、合体がしやすくなる。

### ネクストフィット（Next Fit）

> **「前回探した場所の続きから探す」**

ファーストフィットの弱点（先頭ばかりが細かくなる）を解消する。前回の検索修了位置から探索を再開することで、使用頻度をリスト全体に分散させる。

## 17.4 その他のアプローチ

### 分離リスト（Segregated Lists）

特定サイズの要求専用にメモリプールを用意する。断片化を減らし、割り当て・解放が高速になる。

**スラブアロケータ**（Jeff Bonwick考案）はこのアイデアの優れた実装で、カーネルオブジェクト（ロック、inode等）用に事前に確保されたオブジェクトキャッシュを使う。解放されたオブジェクトを初期化済みの状態で保持し、再利用時の初期化コストを省く。

> 💡 **スラブアロケータ**は、「同じサイズのモノを大量に使うなら、専用の在庫棚を用意する」という発想。例えば工場で同じ部品を繰り返し使うとき、毎回ゼロから作るより「洗浄済み部品」を棚にストックしておく方が効率的だ。

### バディ割り当て（Buddy Allocation）

空きメモリを2の冪乗サイズのブロックで管理する。

![](./images/fig17_7_4.PNG)

要求があるとブロックを再帰的に二等分し、十分小さいブロックを返す。解放時は「バディ」（対になるブロック）が空いていればマージし、再帰的に大きなブロックに戻す。

バディのアドレスは1ビットだけ異なるため、判定が非常に高速。ただし、2の冪乗サイズしか返せないため**内部断片化**が発生する。

### その他

現代のアロケータはバランス木などの高度なデータ構造を使う。マルチプロセッサ環境向けには、Hoardやjemallocなどスケーラブルなアロケータが開発されている。

## 17.5 まとめ

メモリアロケータの設計は多くのトレードオフを伴う。高速でスケーラブルかつ空間効率の良いアロケータの構築は、現代のコンピュータシステムにおける継続的な課題だ。ワークロードの特性を理解するほど、より効率的なアロケータを設計できる。
