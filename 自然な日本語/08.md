# 8. スケジューリング：マルチレベルフィードバックキュー（MLFQ）

MLFQは、最も広く使われているスケジューリング手法の1つだ。1962年にCorbatoらによって発明され、その後長年にわたり改良されてきた。

MLFQは2つの目標を同時に達成しようとする。

1. **ターンアラウンドタイムの最適化**（短いジョブを先に実行）
2. **応答時間の最小化**（インタラクティブなジョブをすばやく実行）

問題は、OSがジョブの実行時間を事前に知らないこと。MLFQは**過去の振る舞いから将来を予測**することで、この問題を解決する。

## 8.1 基本ルール

MLFQは複数のキューを持ち、各キューに異なる優先度が割り当てられている。

- **ルール1**: 優先度(A) > 優先度(B) なら、Aを実行（Bは実行しない）
- **ルール2**: 優先度(A) = 優先度(B) なら、AとBをラウンドロビンで実行

![](../08/img/fig8_1.PNG)

MLFQの特徴は、**観察された振る舞いに基づいて優先度を動的に変更する**こと。

- キーボード入力待ちなどでCPUを頻繁に手放すジョブ → 優先度を高く維持（インタラクティブ）
- CPUを長時間使い続けるジョブ → 優先度を下げる（バッチ処理）

## 8.2 優先度の変更方法（初回の試み）

- **ルール3**: 新しいジョブは最高優先度のキューに配置
- **ルール4a**: タイムスライスを使い切ったら、優先度を1つ下げる
- **ルール4b**: タイムスライスを使い切る前にCPUを手放したら、優先度を維持

### 例1：長時間実行ジョブ

![](../08/img/fig8_2.PNG)

最高優先度（Q2）で開始し、タイムスライスを消費するたびにキューが下がっていく。

### 例2：短いジョブの到着

![](../08/img/fig8_3.PNG)

長時間ジョブAが最低キューで動作中にBが到着。Bは最高キューに入り、短時間で完了する。MLFQはSJFのように振る舞う。

### 例3：I/Oを伴うジョブ

![](../08/img/fig8_4.PNG)

I/Oが多いインタラクティブジョブBは、タイムスライスを使い切る前にCPUを手放すため、高い優先度を維持する。

### 初期MLFQの問題点

1. **飢餓**: インタラクティブジョブが多すぎると、長時間実行ジョブにCPU時間が回らない
2. **ゲーミング**: タイムスライスの直前にわざとI/Oを発行して高優先度を維持する悪用が可能
3. **振る舞いの変化**: CPUバウンドだったジョブがインタラクティブに変わっても、低い優先度のまま

## 8.3 改善策1：優先度ブースト

飢餓を防ぐために、定期的にすべてのジョブを最高優先度キューに戻す。

- **ルール5**: 一定期間Sの後、すべてのジョブを最上位キューに移動

![](../08/img/fig8_5.PNG)

これにより以下が実現する。

- 長時間実行ジョブも定期的にCPU時間を得られる（飢餓の防止）
- CPUバウンドからインタラクティブに変わったジョブにも対応できる

**Sの設定は難しい**。大きすぎると飢餓が起き、小さすぎるとインタラクティブジョブが十分なCPUを得られない。

## 8.4 改善策2：より正確なアカウンティング

ゲーミングを防ぐために、ルール4a/4bを統合する。

- **ルール4（改訂版）**: あるキューレベルでの累積CPU使用量がタイムスライスを超えたら、優先度を下げる（CPUを何回手放したかは無関係）

![](../08/img/fig8_6.PNG)

タイムスライスの直前にI/Oを発行しても、累積時間でカウントされるためゲーミングを防止できる。

## 8.5 チューニングとその他の課題

MLFQのパラメータ設定には多くの選択肢がある。

- キューの数
- 各キューのタイムスライスの長さ
- 優先度ブーストの頻度

![](../08/img/fig8_7.PNG)

一般的な設定例：

- **高優先度キュー**: 短いタイムスライス（例：10ms）→ インタラクティブジョブ向け
- **低優先度キュー**: 長いタイムスライス（例：100ms）→ CPUバウンドジョブ向け

Solarisでは60のキューレベルを持ち、タイムスライスは20ms（最高）〜数百ms（最低）まで変化する。FreeBSDは数式を使って優先度を計算し、CPU使用量の減衰を考慮している。

## 8.6 まとめ

### MLFQの最終ルール

| ルール | 内容 |
|---|---|
| ルール1 | 優先度(A) > 優先度(B) → Aを実行 |
| ルール2 | 優先度(A) = 優先度(B) → ラウンドロビンで実行 |
| ルール3 | 新しいジョブは最高優先度キューに配置 |
| ルール4 | 累積CPU使用量がタイムスライスを超えたら優先度を下げる |
| ルール5 | 定期的にすべてのジョブを最上位キューに移動 |

MLFQが優れている理由：

- ジョブの実行時間に関する事前知識が不要
- 実際の振る舞いを観察して優先度を動的に調整
- 短いインタラクティブジョブには素早い応答（SJF/STCF的な振る舞い）
- 長いCPUバウンドジョブにも公平にCPU時間を配分

BSD UNIX、Solaris、Windows NTなど、多くのOSがMLFQの一形態を基本スケジューラとして採用している。
