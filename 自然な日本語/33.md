# 33. イベントベースの並行性（Event-based Concurrency）

これまで並行性と言えばスレッドを使う方法を見てきたが、それだけが唯一の手段ではない。GUIアプリケーションや一部のインターネットサーバでは、**イベントベースの並行処理**というスタイルが広く使われている。node.jsのような現代的なサーバーサイドフレームワークも、この手法に基づいている。

イベントベースのアプローチが解決する問題は2つある。第1に、マルチスレッドにおける並行性の管理が難しいこと（ロック不足、デッドロックなど）。第2に、スレッドベースではスケジューリングの制御をOSに任せるため、アプリケーション側で最適化しにくいことだ。

> **CRUX: スレッドなしで並行サーバーを構築するには**
> スレッドを使わずに並行性を制御し、マルチスレッドの問題を回避できるか？

## 33.1 基本アイデア：イベントループ

イベントベースの並行処理は非常にシンプルだ。イベントの発生を待ち、その種類を判別して処理する。それだけだ。

```c
while (1) {
    events = getEvents();
    for (e in events)
        processEvent(e);
}
```

各イベントを処理するコードが**イベントハンドラ**だ。ハンドラがイベントを処理している間、それがシステム内で唯一のアクティビティとなる。つまり、次に処理するイベントの決定がスケジューリングそのものだ。これがイベントベースアプローチの基本的な利点——スケジューリングの明示的な制御——を提供する。

## 33.2 重要なAPI：`select()`（または`poll()`）

イベントを受け取るには、`select()`や`poll()`システムコールを使う。着信I/Oの有無を確認するためのAPIだ。

```c
int select(int nfds,
    fd_set *restrict readfds,
    fd_set *restrict writefds,
    fd_set *restrict errorfds,
    struct timeval *restrict timeout);
```

`select()`は、指定されたファイルディスクリプタのセットを調べ、読み込み準備完了・書き込み準備完了・エラー状態を確認する。戻り時には、準備ができたディスクリプタのサブセットに置き換え、その合計数を返す。

> **ASIDE: ブロッキング vs ノンブロッキング**
> ブロッキング（同期）インターフェースは、すべての処理が完了するまで戻らない。ノンブロッキング（非同期）インターフェースは即座に戻り、処理はバックグラウンドで完了する。イベントベースのアプローチでは、ブロッキングがすべての進捗を停止させるため、ノンブロッキングが不可欠だ。

タイムアウト引数をNULLにすれば、ディスクリプタが準備完了になるまで無期限にブロックする。0にすれば即座に戻る。

## 33.4 シンプルな理由：ロック不要

単一CPUのイベントベースアプリケーションでは、一度に1つのイベントしか処理しないため、ロックの取得・解放が不要だ。スレッドに中断される心配がないので、並行性のバグが根本的に発生しない。

> **TIP: イベントベースサーバーではブロックするな**
> イベントベースサーバーはきめ細かなスケジューリング制御を提供するが、ブロッキング呼び出しを行うとその制御を失う。

## 33.5 問題：ブロッキングシステムコール

HTTPリクエストを処理する際、ディスクからファイルを読むために`open()`や`read()`を呼ぶ必要がある。スレッドベースなら、I/O待ちの間に他のスレッドが実行できる。しかしイベントベースでは、メインイベントループしか存在しないため、ブロッキング呼び出しがサーバー全体を停止させてしまう。

## 33.6 解決策：非同期I/O

多くのOSは**非同期I/O**のインターフェースを提供している。I/Oリクエストを発行してすぐに制御を返し、完了の確認は後で行う。

```c
int aio_read(struct aiocb *aiocbp);  // 非同期読み取りを発行
int aio_error(const struct aiocb *aiocbp);  // 完了確認
```

`aio_read()`はI/Oを発行して即座に戻る。`aio_error()`でポーリングして完了を確認する。EINPROGRESSが返れば未完了、0なら完了だ。

ポーリングの問題を解決するため、一部のシステムではUNIXシグナルによる割り込みベースの通知も提供している。

> **ASIDE: UNIXシグナル**
> シグナルはプロセスへの通信手段だ。シグナルが配信されるとアプリケーションは現在の処理を中断し、シグナルハンドラを実行する。SIGSEGV（セグメンテーション違反）など、カーネル自身がシグナルを送る場合もある。

非同期I/Oがないシステムでは、ネットワーク処理にはイベント、ディスクI/Oにはスレッドプールを使うハイブリッドアプローチも有効だ。

## 33.7 もう1つの問題：状態管理

イベントベースのコードでは、非同期I/O発行後にI/O完了時に必要な状態を手動で保存しなければならない。スレッドベースならスタックに状態が自然に保持されるが、イベントベースでは**継続（continuation）**というパターンを使う。

たとえば、ファイルディスクリプタfdからの読み取り完了後にソケットディスクリプタsdに書き込む場合、fdとsdの対応をハッシュテーブルなどに記録しておく。ディスクI/O完了時にfdで検索してsdを取得し、書き込みを行う。Adyaらはこれを「手動スタック管理」と呼んでいる。

## 33.8 未解決の困難

1. **マルチコアへの移行。** 複数CPUを活用するには複数のイベントハンドラを並行実行する必要があり、結局ロックが必要になる。
2. **ページングとの統合。** イベントハンドラがページフォルトを起こすと暗黙的にブロックされ、回避が難しい。
3. **コードの保守性。** ルーチンのセマンティクスがノンブロッキングからブロッキングに変わると、イベントハンドラの分割が必要になる。
4. **I/Oの統一性。** ネットワークI/O（`select()`）とディスクI/O（AIO）で異なるAPIを組み合わせる必要がある。

## 33.9 まとめ

イベントベースの並行処理は、スケジューリングの制御をアプリケーション側に持たせる代わりに、状態管理の複雑さやマルチコアとの統合のコストを伴う。スレッドとイベントはそれぞれ長所と短所があり、どちらか一方が決定的に優れているわけではない。両方のアプローチは今後も共存し続けるだろう。
