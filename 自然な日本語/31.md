# 31. セマフォ（Semaphores）

ロックと条件変数の両方が並行性の問題解決に必要だと分かってきた。しかしDijkstraは、セマフォという単一のプリミティブでこの両方をカバーできると考えた。実際、セマフォはロックとしても条件変数としても使える、汎用的な同期機構だ。

> **CRUX: セマフォの使い方**
> セマフォの定義は何か？ロックや条件変数の代わりにどう使えるのか？セマフォからロックと条件変数を構築することは容易か？

## 31.1 セマフォの定義

セマフォは整数値を持つオブジェクトで、`sem_wait()`と`sem_post()`の2つの操作で制御する。使う前に初期値を設定する必要がある。

```c
sem_t s;
sem_init(&s, 0, 1); // 第3引数が初期値
```

![](./img/fig31_2.PNG)

`sem_wait()`は値をデクリメントし、値が負なら呼び出しスレッドをスリープさせる。`sem_post()`は値をインクリメントし、待機スレッドがあれば1つ起こす。セマフォの値が負のとき、その絶対値は待機中のスレッド数に等しい。

## 31.2 バイナリセマフォ（ロック）

セマフォの初期値を1にすると、ロックとして使える。`sem_wait()`でロック取得、`sem_post()`でロック解放だ。

![](./img/fig31_3.PNG)

**スレッド0だけがクリティカルセクションに入る場合：** `sem_wait()`で値が1→0に減り、すぐに戻る。クリティカルセクション終了後、`sem_post()`で値を1に戻す。

**スレッド0がロック中にスレッド1がロックを試みる場合：** スレッド1の`sem_wait()`で値が0→-1に減り、スレッド1はスリープする。スレッド0が`sem_post()`を呼ぶと値が-1→0に戻り、スレッド1が起動してロックを獲得する。

![](./img/fig31_5.PNG)

ロックは「保持」か「非保持」の2状態しかないので、このような使い方を**バイナリセマフォ**と呼ぶ。

## 31.3 順序付けのためのセマフォ

セマフォは、スレッド間のイベント順序付けにも使える。子スレッドの完了を親が待つ場合、セマフォの初期値を**0**に設定する。

![](./img/fig31_6.PNG)

**親が先に実行される場合：** 親の`sem_wait()`で値が0→-1になり、親はスリープ。子が完了して`sem_post()`を呼ぶと値が-1→0に戻り、親が起動する。

**子が先に実行される場合：** 子の`sem_post()`で値が0→1に。親が`sem_wait()`を呼ぶと値が1→0に減るが、値は0以上なのでスリープせずにそのまま返る。

## 31.4 プロデューサ/コンシューマ問題（有界バッファ）

### 最初の試み

emptyとfullの2つのセマフォを導入する。emptyはMAXで、fullは0で初期化する。

![](./img/fig31_9.PNG)
![](./img/fig31_10.PNG)

プロデューサ1つ、コンシューマ1つ、MAX=1なら正しく動く。しかしMAXが大きく、プロデューサが複数いると、`put()`内で競合状態が発生する。2つのプロデューサが同じバッファ位置にデータを書き込んでしまう可能性がある。

### 相互排除の追加

バイナリセマフォ（ミューテックス）を追加してクリティカルセクションを保護する。

![](./img/fig31_11.PNG)

しかしこれはデッドロックを引き起こす。コンシューマが先にミューテックスを取得し、fullのwaitでブロックされると、プロデューサはミューテックスを取得できず、お互いに待ち合う。

### 正しい解法

ミューテックスの範囲を縮小する。emptyやfullのwait/signalはミューテックスの外に出し、`put()`/`get()`のクリティカルセクションだけをミューテックスで囲む。

![](./img/fig31_12.PNG)

これでデッドロックなく正しく動作する有界バッファが完成する。

## 31.5 リーダライタロック

データ構造への読み取りは複数スレッドが同時に行っても安全だが、書き込みは排他的でなければならない。この区別をサポートするのがリーダライタロックだ。

![](./img/fig31_13.PNG)

仕組みはシンプルだ。最初のリーダがロックに入るとき書き込みロックを取得し、最後のリーダが出るとき書き込みロックを解放する。その間、追加のリーダは自由に入れる。ライターは全リーダの退出を待つ必要がある。

ただしこのアプローチには、リーダがライターを飢餓させやすいという問題がある。また、オーバーヘッドが大きいため、シンプルなスピンロックの方が高速な場合もある。

> **TIP: シンプルで愚直なほうが良い場合がある（Hill's Law）**
> 複雑で洗練されたアプローチより、シンプルで高速な実装のほうが勝ることがある。Mark Hillのキャッシュ研究が示すように、「巨大で愚直なほうが良い」のだ。

## 31.6 食事する哲学者問題

Dijkstraが提起した有名な並行性問題だ。テーブルに5人の哲学者が座り、各ペアの間にフォークが1本ある。食事するには左右のフォーク両方が必要だ。

```c
while (1) {
    think();
    getforks();
    eat();
    putforks();
}
```

### 壊れた解法

各フォークをセマフォ（初期値1）にし、左→右の順でフォークを取る。全員が同時に左のフォークを取ると、右のフォークを待って永遠にデッドロックする。

### 解法：依存関係を断ち切る

最後の哲学者（p=4）だけフォーク取得順序を逆（右→左）にする。これで循環待ちが解消され、デッドロックが防げる。

逆に、セマフォの上に条件変数を構築するのは非常に難しい。Windows環境でこれを試みた経験ある並行プログラマたちは様々なバグに遭遇している。

> **TIP: 一般化には慎重に**
> セマフォはロックと条件変数の一般化と見ることもできるが、Lampsonが言うように「一般化しないで、一般化は一般に間違っている」。

## 31.8 まとめ

セマフォは並行プログラムの強力で柔軟なプリミティブだ。バイナリセマフォ（ロック）、順序付け、有界バッファ、リーダライタロック、食事する哲学者問題と、多くの古典的な同期問題をセマフォで解ける。Allen Downeyの「The Little Book of Semaphores」は、さらに深く学ぶための優れた無料リソースだ。
