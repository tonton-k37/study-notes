# 22. 物理メモリを超えて：ポリシー（Beyond Physical Memory: Policies）

メモリに空きが十分ある場合は単純だ。問題は、メモリが不足した場合にどのページを退去させるか。この判断を行うのが**ページ置換ポリシー**である。

## 22.1 キャッシュ管理

メインメモリは仮想メモリページの**キャッシュ**と見なせる。目標は**キャッシュミス（ディスクからのフェッチ）を最小化**すること。

**平均メモリアクセス時間（AMAT）**:

![](../22/img/fig22_1_1.PNG)

```
AMAT = TM + (PMiss × TD)
```

TM = メモリアクセス時間（〜100ns）、TD = ディスクアクセス時間（〜10ms）

例えば、ミス率90%なら1ミリ秒程度だが、ミス率0.1%ならわずか10マイクロ秒になる。ディスクアクセスのコストが非常に高いため、**わずかなミス率の増加でもパフォーマンスに深刻な影響**がある。

## 22.2 最適（OPT）置換ポリシー

Beladyが証明した理論上最良のポリシー。**将来最も遠い時点でアクセスされるページ**を退去させる。

![](../22/img/fig22_1.PNG)

アクセスストリーム: 0,1,2,0,1,3,0,3,1,2,1

- 最初の3アクセスは強制ミス（コールドスタート）
- ページ3が必要になった時、将来のアクセスを見て最も遠いページ2を退去
- ヒット率: 54.5%（強制ミス除外で85.7%）

**問題**: 将来のアクセスパターンを知ることはできないため、実用的ではない。比較の基準として使う。

## 22.3 FIFO（先入れ先出し）

最初にメモリに入ったページを退去させる。実装が非常に簡単。

![](../22/img/fig22_2.PNG)

しかし、ページの「重要さ」を考慮しないため、頻繁にアクセスされるページでも古ければ追い出してしまう。ヒット率は36.4%（強制ミス除外で57.1%）。

**Beladyの異常**: FIFOではキャッシュサイズを増やすとヒット率が**悪化する**ケースがある。

## 22.5 履歴を活用：LRU

**最も長くアクセスされていないページ**を退去させる。過去のアクセスパターンから将来を予測する考え方。

原理: **局所性の原則**
- **時間的局所性**: 最近アクセスしたページは近い将来も再度アクセスされやすい
- **空間的局所性**: あるページがアクセスされたら近隣のページもアクセスされやすい

![](../22/img/fig22_5.PNG)

LRUは最初のページ置換時にページ2を退去（0と1が最近アクセスされたため）。最適と同等のパフォーマンスを示すケースが多い。

関連手法にLFU（最も使用頻度の少ないページを退去）がある。

## 22.6 ワークロード別の比較

### ランダムアクセス（局所性なし）

![](../22/img/fig22_6.PNG)

局所性がない場合、LRU・FIFO・ランダムの差はなく、ヒット率はキャッシュサイズのみで決まる。

### 80-20ワークロード（局所性あり）

![](../22/img/fig22_7.PNG)

参照の80%がページの20%（ホットページ）に集中。LRUがホットページを保持するため、FIFO・ランダムより優れる。

### ループ順アクセス

![](../22/img/fig22_8.PNG)

50ページを順番にアクセスするループ。LRUとFIFOの**最悪ケース**。古いページを追い出すが、ループの性質上それらがすぐ必要になる。49エントリのキャッシュでもヒット率は0%。ランダムの方が良い結果を示す。

## 22.7 LRUの実装の課題

完全なLRUの実装には、**すべてのメモリアクセスで**データ構造を更新する必要があり、非常に高コスト。

ハードウェアで各ページにタイムスタンプを付けても、置換時にすべてのタイムスタンプをスキャンする必要がある（100万ページ以上をスキャン）。

## 22.8 LRUの近似：クロックアルゴリズム

完全なLRUの代わりに、**ユースビット（参照ビット）**を使ったLRUの近似を行う。

### クロックアルゴリズムの動作

1. すべてのページを循環リストに配置
2. 時計の針をページ上に移動
3. ユースビット = 1 → 最近使用された。ビットを0にクリアして次に進む
4. ユースビット = 0 → 最近使用されていない。このページを退去させる

![](../22/img/fig22_9.PNG)

完全なLRUには劣るが、履歴を全く考慮しない手法よりは良い結果を示す。

## 22.9 ダーティページの考慮

退去させるページが**ダーティ（変更済み）**なら、ディスクに書き戻す必要がありコストが高い。**クリーン（未変更）ページ**は書き戻し不要で安価に退去できる。

クロックアルゴリズムに**ダーティビット**を組み込み、未使用かつクリーンなページを優先的に退去させることで効率を上げる。

## 22.10 その他のVMポリシー

### ページ選択ポリシー
- **デマンドページング**: アクセス時に初めてメモリに読み込む
- **プリフェッチ**: 近くのページを先読みする（例：コードページPの次にP+1も読む）

### 書き出しポリシー
- **クラスタリング**: 複数のページをまとめてディスクに書き出し、I/O効率を向上

## 22.11 スラッシング

プロセスのメモリ要求が物理メモリを大幅に超えると、**スラッシング**（絶え間ないページング）が発生し、システムが極端に遅くなる。

対策：
- **アドミッションコントロール**: 一部のプロセスの実行を停止し、残りのプロセスのワーキングセットがメモリに収まるようにする
- **OOMキラー（Linux）**: メモリを大量に消費するプロセスを強制終了

## 22.12 まとめ

ページ置換ポリシーの基本を学んだ。現代システムではクロックアルゴリズムのようなLRU近似が使われる。しかし、メモリとディスクの速度差が拡大する中で、最も重要なのは**ページングの発生自体を減らすこと**（十分なメモリを搭載する）だ。
