# 49. Sunのネットワークファイルシステム（NFS）

分散クライアント/サーバーコンピューティングの初期の応用の一つが分散ファイルシステムだ。複数のクライアントマシンが1台（または数台）のサーバーのデータにネットワーク経由でアクセスする。

![](./img/fig49_1.PNG)

なぜローカルディスクだけではダメか？この構成により**クライアント間のデータ共有**が容易になる。マシンAでファイルを編集し、マシンBで開けば同じファイルが見える。ほかにも集中管理（少数のサーバーからバックアップ）やセキュリティ（サーバーを物理的に安全な場所に配置）の利点がある。

> **CRUX: 分散ファイルシステムをどう構築するか？**

## 49.1 基本アーキテクチャ

クライアントアプリケーションはシステムコール（`open()`、`read()`、`write()`、`close()`など）を**クライアント側ファイルシステム**に発行する。クライアント側FSは必要に応じてサーバーにメッセージを送り、サーバー側FS（ファイルサーバー）がディスクからデータを読み返す。

![](./img/fig49_2.PNG)

アプリケーションにとってはローカルFSとの違いはパフォーマンスくらいで、透過的なアクセスが実現する。

## 49.2 NFSの誕生

Sun Microsystemsが開発し、**オープンプロトコル**として公開した。クローズドなシステムではなく、メッセージフォーマットを公開することで、異なるベンダーがNFSサーバーを独自に開発・競争できる。Oracle/Sun、NetApp、EMC、IBMなど多くの企業がNFSサーバーを提供している。

## 49.3 設計の焦点：シンプルで高速なクラッシュリカバリ

NFSv2の主目標は**シンプルで高速なサーバークラッシュリカバリ**だ。サーバーが落ちると全クライアントが影響を受ける。だから復旧は素早くなければならない。

## 49.4 高速リカバリの鍵：ステートレス

NFSv2はステートレスプロトコルとして設計されている。サーバーは、どのクライアントがどのブロックをキャッシュしているか、どのファイルが何番のディスクリプタで開かれているかを一切把握しない。各プロトコル要求に、リクエスト完了に必要なすべての情報が含まれている。

**ステートフルの問題**：`open()`でファイルディスクリプタfdを返し、`read(fd, ...)`でアクセスする。この共有状態「fdはファイルfooを参照」はサーバーメモリに保持される。サーバーがクラッシュして再起動すると、fdの意味がわからなくなる。リカバリプロトコルが必要。さらに、クライアントがクラッシュした場合、サーバーはファイルの`close()`を受け取れず、リソースリークが起きる。

![](./img/fig49_3.PNG)

ステートレスなら、このような複雑な回復は不要。サーバーは再起動するだけ、クライアントはリクエストを再試行するだけだ。

## 49.5 NFSv2プロトコル

> **CRUX: ステートレスなファイルプロトコルをどう定義するか？**

鍵は**ファイルハンドル**だ。操作対象のファイルやディレクトリを一意に識別する。3つの要素で構成される：

- **ボリューム識別子** — どのファイルシステムか
- **inode番号** — どのファイルか
- **世代番号** — inode番号の再利用で古いハンドルが誤って使われるのを防ぐ

![](./img/fig49_4.PNG)

主要なプロトコルメッセージ：

- **LOOKUP** — ディレクトリハンドル + ファイル名 → ファイルハンドル + 属性を返す
- **READ** — ファイルハンドル + オフセット + バイト数 → データを返す
- **WRITE** — ファイルハンドル + オフセット + データ → 成功コードを返す
- **GETATTR** — ファイルハンドル → 属性（最終変更時刻など）を返す

## 49.6 プロトコルからファイルシステムへ

![](./img/fig49_5.PNG)

クライアント側FSが整数ファイルディスクリプタ→NFSファイルハンドル、現在のファイルポインタのマッピングなど、すべての状態を管理する。`/home/remzi/foo.txt`を開くと、クライアントは3つのLOOKUPを送る：`/`の`home`、`home`の`remzi`、`remzi`の`foo.txt`。

サーバーはリクエスト完了に必要な状態を一切必要としない。

> **TIP: 冪等性は強力だ**
> 操作を複数回実行しても結果が同じ——冪等な操作なら、障害時に単純に再試行できる。

## 49.7 冪等操作によるサーバー障害処理

クライアントが応答を受信しない理由はさまざまだ。ネットワークがメッセージを破棄、サーバーがクラッシュ中、応答が失われた。NFSv2では**すべてのケースで均一に**処理する——単純にリクエストを再試行する。

![](./img/fig49_6.PNG)

ほとんどのNFS操作は冪等だ。LOOKUP、READは情報の読み取りのみ。WRITEも冪等——データ、カウント、正確なオフセットが含まれるため、何度書いても結果は同じ。

一部は完全な冪等性を持たない。すでに存在するディレクトリへのMKDIRは2回目に失敗する。しかしこれは稀なケースだ。

> **TIP: 完璧は良の敵（ヴォルテールの法則）**
> 大部分のケースをカバーすれば良い。全コーナーケースの完璧な処理を追求しすぎると、何も構築できない。

## 49.8 パフォーマンス改善：クライアント側キャッシュ

すべてのリクエストをネットワーク越しに送ると遅い。解決策は**クライアント側キャッシュ**だ。

- **読み取りキャッシュ**：サーバーから読んだデータをクライアントメモリに保持。2回目以降はネットワーク不要
- **書き込みバッファリング**：データをクライアントメモリにバッファし、後でサーバーにフラッシュ。`write()`は即座に返る

## 49.9 キャッシュ一貫性の問題

複数クライアントが同じファイルにアクセスすると2つの問題が発生する。

![](./img/fig49_7.PNG)

### 更新の可視性（Update Visibility）

C2がファイルFを更新しても、ローカルキャッシュにバッファリングしている間はサーバーに反映されない。C3がFを読むと古いバージョンが返る。

### キャッシュの陳腐化（Stale Cache）

C2がサーバーに書き込んだ後も、C1はキャッシュ内の古いバージョンを持ち続ける。

### NFSv2の対策

1. **flush-on-close（close-to-open一貫性）**：ファイルを閉じる時にダーティページをすべてサーバーにフラッシュ。次にどこかのクライアントが開くと最新版が見える

2. **ファイルオープン時のGETATTR**：開く前にサーバーにファイルの属性（最終変更時刻）を問い合わせ、キャッシュ版より新しければキャッシュを無効化

**問題**：GETATTRリクエストの洪水がサーバーを圧倒する。対策として**属性キャッシュ**を導入——属性を一定時間（3秒など）キャッシュし、その間はサーバーに問い合わせない。しかし、この間にファイルが更新されると古いバージョンが返る可能性がある。

## 49.10 NFSキャッシュ一貫性の評価

flush-on-closeは一時ファイルにも適用されるため無駄なサーバー通信が発生する。属性キャッシュはファイルのどのバージョンを取得しているか不確実になる。実装の詳細がユーザーから見えるセマンティクスを定義している興味深い例だ。

## 49.11 サーバー側書き込みバッファリングの影響

NFSサーバーは**書き込みが安定ストレージにコミットされるまでWRITE成功を返してはならない**。

```c
write(fd, a_buffer, size); // ブロック1をaで埋める
write(fd, b_buffer, size); // ブロック2をbで埋める
write(fd, c_buffer, size); // ブロック3をcで埋める
```

2番目の書き込みがメモリにバッファリングされた状態で成功を返し、サーバーがクラッシュすると：

```
aaaa...aaaa
yyyy...yyyy  ← 古いデータが残る！
cccc...cccc
```

クライアントは3つとも成功と認識しているのに、データが混ざる。これを防ぐため、書き込みを安定ストレージにコミットしてから成功を返す必要がある。

NetAppなどの企業はバッテリーバックアップメモリに書き込むことで、永続性を保証しつつ高速な応答を実現した。

## 49.12 まとめ

NFSはサーバー障害からのシンプルかつ高速なリカバリを中心に設計されている。ステートレスプロトコルにより、操作の冪等性がクライアントのリクエスト再試行を安全にする。クライアント側キャッシュはパフォーマンスを向上させるが、キャッシュ一貫性問題を生む。サーバーへの書き込みは安定ストレージへのコミットが必須——さもなくばデータ損失のリスクがある。
