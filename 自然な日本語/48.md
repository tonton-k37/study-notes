# 48. 分散システム（Distributed Systems）

分散システムは世界の姿を変えた。ウェブブラウザがどこかのサーバーに接続するとき、それは単純なクライアント/サーバー型の分散システムだ。GoogleやFacebookのような大規模サービスの裏側では、数千台のマシンが協調して動いている。

分散システム構築の中心課題は**障害**だ。マシン、ディスク、ネットワーク、ソフトウェア——すべてが時々故障する。しかし、ユーザには「失敗していない」ように見せたい。故障するパーツから動くシステムを作る——これが分散システムの核心的な美しさだ。

> **CRUX: コンポーネントが故障しても動くシステムをどう構築するか？**

## 48.1 通信の基礎

現代のネットワークの根本的な教えは、**通信は本質的に信頼できない**ということだ。広域インターネットでもローカルの高速ネットワークでも、パケットは失われたり、破損したり、宛先に届かないことがある。

パケットロスの原因は多様だ。電気的問題によるビット反転、ネットワーク機器の故障、ケーブルの切断。しかし最も根本的なのは**バッファ溢れ**だ。ルータやエンドホストの到着パケットがメモリ容量を超えると、パケットを捨てるしかない。

> **TIP: 通信は本質的に信頼できない**
> パケットの破損、リンクやマシンのダウン、バッファ不足——すべて同じ結果に行き着く。パケットが届かないことがある。信頼できないネットワークの上に信頼性のあるサービスを構築するには、パケットロスに対処する技術が必要だ。

## 48.2 信頼性のない通信層

最も単純なアプローチ——パケットロスに対処しない。アプリケーション自身がパケットロスを処理できる場合、信頼性の低いメッセージング層を直接使えると便利な場面がある。

その好例が**UDP/IP**だ。UDPはパケットロスを送信者に通知しないが、チェックサムによるパケット破損の検出は行う。

![](./img/fig48_1.PNG)
![](./img/fig48_2.PNG)

> **TIP: 完全性にはチェックサムを使え**
> チェックサムは破損を素早く効果的に検出する一般的な手法だ。送信前にメッセージのチェックサムを計算し、メッセージと共に送る。受信側で再計算して一致すれば破損なしと判断できる。

## 48.3 信頼性のある通信層

信頼できる通信には、パケットロスに対処するメカニズムが必要だ。基本は**確認応答（ACK）**と**タイムアウト/リトライ**だ。

![](./img/fig48_3.PNG)

1. 送信者がメッセージを送り、受信者がACKを返す
2. ACKが一定時間内に届かなければ、送信者はメッセージが失われたと判断して再送する

![](./img/fig48_4.PNG)

しかし、元のメッセージではなく**ACKが失われた**場合はどうか？送信者は再送するが、受信者は同じメッセージを2回受け取ってしまう。

![](./img/fig48_5.PNG)

この重複を検出するために**シーケンスカウンタ**を使う。送信者と受信者がカウンタ値で合意し、メッセージにカウンタ値を付ける。受信者は期待値と一致するメッセージだけをアプリケーションに渡し、重複メッセージにはACKだけ返してアプリケーションには渡さない。

最も広く使われる信頼性のある通信層が**TCP/IP**だ。輻輳制御、複数の未解決リクエスト、膨大な最適化を含む。

> **TIP: タイムアウト値の設定は慎重に**
> 小さすぎると不必要な再送——リソースの浪費。大きすぎると再送が遅すぎてパフォーマンス低下。多数のクライアントからサーバーへの送信では、パケットロスはサーバー過負荷の指標かもしれない。**指数バックオフ**で再送間隔を2倍ずつ増やし、過負荷の悪循環を回避する。

## 48.4 通信の抽象化

基本的なメッセージング層の上に、分散システムではどんな通信抽象化を使うべきか？

### 分散共有メモリ（DSM）

OSの仮想メモリを拡張し、異なるマシン上のプロセスが大きな仮想アドレス空間を共有する。ローカルにないページへのアクセスはページフォルトを発生させ、リモートマシンからフェッチする。

**問題点**：障害処理が非常に困難。マシンがクラッシュするとそのマシンのページが失われ、アドレス空間に穴が開く。パフォーマンスも不均一——一部のメモリアクセスは安価、他は高価なリモートフェッチを引き起こす。今日、DSMで信頼できる分散システムを構築する人はいない。

## 48.5 リモートプロシージャコール（RPC）

プログラミング言語の抽象化の方がはるかに実用的だ。最も支配的なのが**RPC（Remote Procedure Call）**だ。リモートマシンでコードを実行するプロセスを、ローカル関数呼び出しと同じくらい簡単にする。

### スタブジェネレータ

インターフェース定義から、引数のパッキング/アンパッキングを自動化するコードを生成する。

```c
interface {
    int func1(int arg1);
    int func2(int arg1, int arg2);
};
```

**クライアントスタブ**の動作：
1. メッセージバッファを作成
2. 関数IDと引数をバッファにパック（**マーシャリング/シリアライゼーション**）
3. RPCサーバーにメッセージ送信
4. 応答を待つ
5. 戻り値をアンパック（**アンマーシャリング/デシリアライゼーション**）
6. 呼び出し元に返す

**サーバー側**：メッセージのアンパック → 実際の関数呼び出し → 結果のパック → 応答送信。

サーバーの並行性には**スレッドプール**が一般的。起動時に有限のスレッドを作成し、到着メッセージをワーカースレッドにディスパッチする。

### ランタイムライブラリ

RPCシステムの主要課題：

- **サービスの発見（ネーミング）**：ホスト名/IPアドレスとポート番号でRPCサービスを特定
- **トランスポートプロトコルの選択**：TCPは信頼性があるが余分なACKメッセージのオーバーヘッド。多くのRPCパッケージはUDP上に構築し、RPC層自身がタイムアウト/リトライを実現
- **バイトオーダー**：ビッグエンディアンとリトルエンディアンの変換。XDR（eXternal Data Representation）など
- **非同期RPC**：リクエスト送信後すぐに戻り、クライアントは他の作業を行う。後で結果を取得

> **ASIDE: エンドツーエンド原則**
> システム内で最終的に特定の機能を真に実装できるのはエンドポイント（アプリケーション）だけという原則。ファイル転送の信頼性は通信層だけでは保証できない——送信前のメモリ破損やディスク書き込みエラーには対処できない。エンドツーエンドのチェックサム照合が必要だ。ただし低レベル層の信頼性機構も性能最適化として有用。

## 48.6 まとめ

分散システムの中心課題は障害処理だ。通信はシステムの核心を形成し、その一般的な抽象化がRPCだ。RPCパッケージはタイムアウト/リトライ、ACK、シーケンス番号を処理し、ローカル関数呼び出しを模倣するサービスを提供する。

実際にRPCを理解する最良の方法は自分で使うことだ。SunのRPC（rpcgen）は古典的。GoogleのgRPCやApache Thriftは現代的な選択肢だ。
