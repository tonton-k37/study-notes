B-Treeはsqliteがテーブルとインデックスの両方を表現するために使うデータ構造であり、非常に中心的な概念だ。この記事ではデータ構造の紹介だけを行い、コードは書かない。

なぜツリーがデータベースに適したデータ構造なのか？

- 特定の値の検索が高速（対数時間）
- 見つけた値の挿入・削除が高速（リバランスに要する時間はほぼ定数）
- 値の範囲を走査するのが高速（ハッシュマップと違って）

B-Treeは二分木とは異なるものだ（「B」はおそらく発明者の名前に由来するが、「balanced（バランス）」の略かもしれない）。B-Treeの例を示す：

{% include image.html url="assets/images/B-tree.png" description="B-Treeの例 (https://en.wikipedia.org/wiki/File:B-tree.svg)" %}

二分木と違い、B-Treeの各ノードは2つ以上の子を持てる。各ノードは最大m個の子を持つことができ、mをツリーの「次数」と呼ぶ。ツリーのバランスを保つため、ノードは少なくともm/2個（切り上げ）の子を持つ必要がある。

例外：
- リーフノードは子を0個持つ
- ルートノードはm個未満の子を持てるが、少なくとも2個は必要
- ルートノードがリーフノード（唯一のノード）の場合は、子が0個でもよい

上の図はB-Treeで、SQLiteがインデックスの保存に使う。テーブルの保存にはB+ treeという変種を使う。

|                                 | B-tree           | B+ tree               |
|---------------------------------|------------------|-----------------------|
| 読み方                          | 「ビーツリー」     | 「ビープラスツリー」     |
| 用途                            | インデックス       | テーブル               |
| 内部ノードにキーを保存           | する              | する                   |
| 内部ノードに値を保存             | する              | しない                 |
| ノードあたりの子の数             | 少ない            | 多い                   |
| 内部ノードとリーフノードの構造   | 同じ              | 異なる                 |

インデックスの実装に取りかかるまでは、B+ treeだけを扱うが、単にB-treeまたはbtreeと呼ぶことにする。

子を持つノードは「内部」ノードと呼ばれる。内部ノードとリーフノードは異なる構造を持つ：

| 次数mのツリーの場合...   | 内部ノード               | リーフノード         |
|--------------------------|--------------------------|----------------------|
| 保存するもの             | キーと子へのポインタ      | キーと値             |
| キーの数                 | 最大m-1個                | 入るだけ             |
| ポインタの数             | キーの数 + 1             | なし                 |
| 値の数                   | なし                     | キーの数と同じ       |
| キーの役割               | ルーティングに使用        | 値とペアになる       |
| 値を保存するか？          | しない                   | する                 |

B-treeに要素を挿入する際にどのように成長するか、例を通じて見てみよう。シンプルにするため、ツリーの次数は3とする。つまり：

- 内部ノードあたり最大3つの子
- 内部ノードあたり最大2つのキー
- 内部ノードあたり少なくとも2つの子
- 内部ノードあたり少なくとも1つのキー

空のB-treeは単一のノード、ルートノードだけを持つ。ルートノードはキー/値ペアが0個のリーフノードとして始まる：

{% include image.html url="assets/images/btree1.png" description="空のbtree" %}

キー/値ペアをいくつか挿入すると、ソートされた順序でリーフノードに保存される。

{% include image.html url="assets/images/btree2.png" description="1ノードのbtree" %}

リーフノードの容量が2つのキー/値ペアだとしよう。もう1つ挿入すると、リーフノードを分割してペアの半分をそれぞれのノードに入れる必要がある。両方のノードは新しい内部ノードの子になり、この内部ノードが新しいルートノードになる。

{% include image.html url="assets/images/btree3.png" description="2階層のbtree" %}

内部ノードは1つのキーと子ノードへの2つのポインタを持つ。5以下のキーを探す場合は左の子を見る。5より大きいキーを探す場合は右の子を見る。

次にキー「2」を挿入しよう。まず、もし存在するならどのリーフノードにあるかを調べ、左のリーフノードにたどり着く。ノードがいっぱいなので、リーフノードを分割し、親ノードに新しいエントリを作成する。

{% include image.html url="assets/images/btree4.png" description="4ノードのbtree" %}

キーを追加し続けよう。18と21。再び分割が必要になるが、親ノードにキー/ポインタペアの空きがない。

{% include image.html url="assets/images/btree5.png" description="内部ノードに空きがない" %}

解決策は、ルートノードを2つの内部ノードに分割し、新しいルートノードをそれらの親として作成すること。

{% include image.html url="assets/images/btree6.png" description="3階層のbtree" %}

ツリーの深さが増えるのはルートノードを分割する時だけだ。すべてのリーフノードは同じ深さを持ち、ほぼ同じ数のキー/値ペアを保持するため、ツリーはバランスを保ち、検索が高速であり続ける。

キーの削除についてはここでは触れず、挿入を実装した後で議論する。

このデータ構造を実装する際、各ノードは1つのページに対応する。ルートノードはページ0に存在する。子ポインタは単に子ノードが格納されているページ番号になる。
